Cryptographically Verifiable Intent Chain for AI Agent Content Provenance

Abstract

This document defines the intent_chain claim as a companion to the actor_chain claim defined in [[!I-D.draft-mw-spice-actor-chain]]. While the actor chain addresses delegation provenance (WHO delegated to whom), the intent chain addresses content provenance (WHAT was produced and HOW it was transformed).¶

In AI agent workflows, content flows through multiple processing stages including AI agents and filters. The intent chain provides a cryptographically verifiable, tamper-evident record of this content journey. The full intent chain is stored as ordered logs, with only the merkle root included in the OAuth token for efficiency.¶

Together, the actor chain and intent chain provide complete governance for autonomous AI agent systems, addressing Spoofing, Tampering, Repudiation, and Elevation of Privilege threats in the STRIDE threat model.¶

1. Introduction

1.1. The Problem: Content Provenance Gap

The Actor Chain extension to [RFC8693] addresses the Delegation Auditability Gap by providing cryptographic proof of the actual delegation path between AI agents. However, it does not address a complementary gap: Content Provenance.¶

In AI agent workflows, content flows through multiple processing stages:¶

Agent A -> Filter -> Filter -> Agent B -> Filter -> Agent C -> Tool
¶

Each stage may transform the content. AI agent outputs are inherently non-deterministic and cannot be trusted without validation. Filters (both AI-based and rule-based) transform this content before it reaches the next stage.¶

For complete governance, systems require proof of:¶

  • What each AI agent originally produced (raw output)¶
  • How each filter transformed the content¶
  • Whether transformations were deterministic (reproducible) or non-deterministic (AI-based)¶
  • The complete chain of content transformations within a session¶

Without this proof, agents can repudiate their outputs, and the content transformation history cannot be reconstructed for audit or dispute resolution.¶

1.2. Relationship to Actor Chain

This specification is designed as a companion to the actor chain, not a replacement:¶

Table 1
Specification Scope STRIDE Coverage
Actor Chain ([[!I-D.draft-mw-spice-actor-chain]]) WHO delegated to whom Spoofing, Repudiation, Elevation of Privilege
Intent Chain (this document) WHAT was produced and transformed Repudiation, Tampering

The two chains serve different purposes and can be used independently or together:¶

  • Actor Chain Only: Authorization decisions, access control¶
  • Intent Chain Only: Content audit, debugging, filter validation¶
  • Combined: Full governance, dispute resolution, regulatory compliance¶

1.3. Design Goals

The intent chain is designed with the following goals:¶

  1. Content Provenance: Cryptographic proof of what each agent produced¶
  2. Transformation Tracking: Record of how filters modified content¶
  3. Tamper Evidence: Merkle tree structure prevents undetected modification¶
  4. Efficiency: Only merkle root in token; full chain in logs¶
  5. Scalability: Append-only logs scale horizontally¶
  6. Modularity: Usable independently or with actor chain¶
  7. Standards Alignment: Compatible with OAuth 2.0, JWT, SPIFFE¶

2. Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14 [[RFC2119]] [[RFC8174]] when, and only when, they appear in all capitals, as shown here.¶

Intent Chain:
An ordered sequence of Intent Chain Entries representing the complete content journey from originating agent through filters to final output within a session.¶
Intent Chain Entry:
A record identifying a single content transformation, including the agent identity, entry type, content hashes, and cryptographic signature.¶
AI Agent:
An autonomous decision-maker that produces content and can delegate authority. AI agents appear in both the actor chain (for delegation) and the intent chain (for output provenance).¶
Non-Deterministic Filter:
A processor (typically AI-based) whose output cannot be reproduced from its input. Examples include AI guardrails, LLM-based content rewriters, and semantic classifiers. Both input and output MUST be signed.¶
Deterministic Filter:
A processor whose output can be reproduced from its input and rules. Examples include schema validators, regex sanitizers, and bounds checkers. Output can be re-derived for verification.¶
Intent Root:
The merkle root hash of the complete intent chain, included in the OAuth token.¶
Intent Registry:
An append-only ordered log storing the full intent chain entries, partitioned by session.¶
Actor Registry:
A database storing mutable state including capabilities and revocation status.¶

3. Architecture Overview

3.1. Three-Layer Governance Model

The governance model consists of three layers:¶

  • Session: Root of trust and lifecycle management. Initiated by human approval or system authorization. Contains subject, expiry, and approval reference.¶
  • Actor Chain (WHO): Contains AI agents only. Full chain stored in token. Addresses Spoofing, Repudiation, and Elevation of Privilege. Defined in [[!I-D.draft-mw-spice-actor-chain]].¶
  • Intent Chain (WHAT): Contains AI agents and filters. Full chain stored in ordered logs; merkle root in token. Addresses Repudiation and Tampering. Defined in this document.¶

3.2. Session as Root of Trust

Every governance chain traces back to a session. The session provides:¶

  • Origin: Who or what initiated the workflow¶
  • Lifecycle: When the session expires¶
  • Revocation Point: Revoking session invalidates all activity¶
  • Audit Boundary: Session defines the unit of audit¶

Session information is captured in standard OAuth token claims:¶

Table 2
Claim Purpose
sub Session subject (human or system initiator)
jti Session identifier
iat Session start time
exp Session expiry time

3.3. Actor Chain (WHO) - Reference

The actor chain is defined in [[!I-D.draft-mw-spice-actor-chain]]. Key properties:¶

  • Contains AI agents only (autonomous decision-makers)¶
  • Tracks delegation of authority¶
  • Full chain included in token¶
  • Addresses Spoofing, Repudiation, Elevation of Privilege¶

This document assumes familiarity with the actor chain specification.¶

3.4. Intent Chain (WHAT) - This Document

The intent chain is defined in this document. Key properties:¶

  • Contains AI agents AND filters¶
  • Tracks content production and transformation¶
  • Merkle root in token; full chain in ordered logs¶
  • Addresses Repudiation and Tampering¶

3.5. STRIDE Threat Model Coverage

Table 3
Threat Mitigation Component
S - Spoofing Cryptographic identity, signed chain entries Actor Chain
T - Tampering Merkle tree integrity, append-only logs Intent Chain
R - Repudiation Signed delegation (Actor) + Signed outputs (Intent) Both
I - Information Disclosure Selective disclosure (SD-JWT) Both (optional)
D - Denial of Service Session expiry, rate limits Session + Infrastructure
E - Elevation of Privilege Scope attenuation, policy enforcement Actor Chain

4. Intent Chain Definition

4.1. Entry Types

The intent chain contains three types of entries:¶

Table 4
Entry Type Sub Determinism Signing Requirement
Agent Output AI Agent Non-deterministic Sign input_hash + output_hash
Non-Deterministic Filter AI-based filter Non-deterministic Sign input_hash + output_hash
Deterministic Filter Rule-based filter Deterministic Sign input_hash + rule_id (output_hash recorded; output reproducible)

All entry types REQUIRE both input_hash and output_hash. This uniform structure ensures that every consecutive pair satisfies entry[i].output_hash == entry[i+1].input_hash, creating a complete content provenance chain. The cost is approximately 40 bytes per entry in the ordered logs — not in the token itself, which carries only the merkle root regardless of entry count.¶

4.2. Agent Output Entries

Agent output entries record the raw output produced by an AI agent before any filtering.¶

Purpose: Prove what the agent originally produced; prevent repudiation of agent outputs.¶

Properties:¶

  • Sub is an AI agent (also appears in actor chain)¶
  • Output is non-deterministic (cannot be reproduced)¶
  • input_hash MUST be recorded (hash of the input received by the agent)¶
  • output_hash MUST be signed by the agent¶

Structure:¶

{
  "type": "agent_output",
  "sub": "spiffe://example.com/agent/orchestrator",
  "input_hash": "sha256:fff000...",
  "output_hash": "sha256:abc123...",
  "iat": 1700000010,
  "intent_digest": "sha256:...",
  "intent_sig": "eyJhbGci..."
}
¶

4.3. Non-Deterministic Filter Entries

Non-deterministic filter entries record transformations by AI-based filters whose output cannot be reproduced from the input.¶

Examples:¶

  • AI guardrails (Llama Guard, NeMo Guardrails)¶
  • LLM-based content rewriters¶
  • Semantic classifiers¶
  • Context-aware sanitizers¶

Properties:¶

  • Producer is an AI-based filter¶
  • Output cannot be reproduced from input¶
  • BOTH input_hash AND output_hash MUST be signed¶

Structure:¶

{
  "type": "non_deterministic_filter",
  "sub": "spiffe://example.com/filter/ai-guardrail",
  "filter_version": "v2.1",
  "input_hash": "sha256:abc123...",
  "output_hash": "sha256:def456...",
  "model_info": {
    "model": "llama-guard-3",
    "categories": ["violence", "pii", "prompt_injection"]
  },
  "iat": 1700000015,
  "intent_digest": "sha256:...",
  "intent_sig": "eyJhbGci..."
}
¶

4.4. Deterministic Filter Entries

Deterministic filter entries record transformations by rule-based filters whose output can be reproduced from the input and rules.¶

Examples:¶

  • Schema validators (JSON Schema)¶
  • Regex sanitizers (XSS removal)¶
  • Bounds checkers (amount limits)¶
  • PII redactors (pattern-based)¶

Properties:¶

  • Sub is a rule-based filter¶
  • Output CAN be reproduced from input + rules¶
  • input_hash and rule_id MUST be recorded and signed¶
  • output_hash MUST be recorded (signed via intent_digest)¶
  • Output can be re-derived for independent verification¶

Structure:¶

{
  "type": "deterministic_filter",
  "sub": "spiffe://example.com/filter/schema-validator",
  "filter_version": "v1.0",
  "input_hash": "sha256:def456...",
  "output_hash": "sha256:ghi789...",
  "rule_id": "ticket-schema-v2",
  "rule_hash": "sha256:rrr...",
  "transform_applied": {
    "fields_validated": ["title", "priority", "amount"],
    "fields_modified": ["priority"],
    "modification": {
      "priority": {
        "from": "critical",
        "to": "medium",
        "reason": "bounds_exceeded"
      }
    }
  },
  "reproducible": true,
  "iat": 1700000016
}
¶

4.5. Entry Structure

All intent chain entries share common fields:¶

Table 5
Field Type Required Description
type string REQUIRED Entry type: agent_output, non_deterministic_filter, deterministic_filter
sub string REQUIRED SPIFFE ID of the agent or filter
input_hash string REQUIRED SHA-256 hash of the input content
output_hash string REQUIRED SHA-256 hash of the output content
iat number REQUIRED Timestamp when entry was created
intent_digest string REQUIRED Hash of the canonically serialized entry for merkle leaf computation
intent_sig string CONDITIONAL Signature over intent_digest (required for non-deterministic)

4.5.1. intent_digest Computation

The intent_digest field is computed as the SHA-256 hash of the canonically serialized entry, excluding the intent_digest and intent_sig fields themselves. This hash serves as the leaf node in the merkle tree.¶

For an entry E with fields {type, sub, output_hash, iat, ...}:¶

intent_digest = SHA-256(canonical_json(E \ {intent_digest, intent_sig}))
¶

Where canonical_json follows JSON Canonicalization Scheme [JCS] to ensure deterministic serialization.¶

The intent_sig (when REQUIRED) is computed over the intent_digest value using the agent's private key:¶

intent_sig = Sign(agent_key, intent_digest)
¶

This two-step process ensures that: (a) the digest is stable and independent of signature ordering, and (b) the signature covers all content-relevant fields of the entry.¶

Additional fields by entry type:¶

Table 6
Field Entry Types Description
filter_version Filters Version of filter
rule_id Deterministic Identifier of rule applied
rule_hash Deterministic Hash of rule definition
model_info Non-deterministic AI model information
transform_applied Filters Details of transformation
reproducible Deterministic Boolean indicating reproducibility

5. Storage Architecture

5.1. Actor Registry (Database)

The actor registry stores mutable state required for authorization decisions.¶

Contents:¶

  • Capability contracts¶
  • Revocation status¶
  • Agent metadata¶
  • Scope definitions¶

Properties:¶

  • Mutable (status can change)¶
  • Strong consistency required¶
  • Point queries by ID¶
  • Updates for revocation¶

Implementation Options: DynamoDB, PostgreSQL, CockroachDB.¶

Schema Example:¶

CREATE TABLE capabilities (
    capability_id   VARCHAR PRIMARY KEY,
    agent_spiffe_id VARCHAR NOT NULL,
    agent_type      VARCHAR NOT NULL,
    owner           VARCHAR NOT NULL,
    allowed_tools   JSONB,
    resource_scope  JSONB,
    status          VARCHAR DEFAULT 'ACTIVE',
    expires_at      TIMESTAMP,
    signature       TEXT,
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_capabilities_agent ON capabilities(agent_spiffe_id);
CREATE INDEX idx_capabilities_status ON capabilities(status);
¶

5.2. Intent Registry (Ordered Logs)

The intent registry stores immutable intent chain entries as ordered logs.¶

Contents:¶

  • Agent output entries¶
  • Filter entries (non-deterministic and deterministic)¶
  • Timestamps and signatures¶

Intent registry entries MUST NOT contain OAuth tokens, bearer credentials, or signing keys. Entries contain only content hashes, metadata, agent identities, and entry-level signatures. The token references the registry via the intent_registry claim; the registry MUST NOT store or reference the token itself.¶

Properties:¶

  • Append-only (immutable)¶
  • Ordered by offset within session¶
  • Partitioned by session ID¶
  • Eventual consistency acceptable¶

Implementation Options: Apache Kafka, AWS Kinesis, Apache Pulsar, SCITT transparency log.¶

Log Structure:¶

{
  "session_id": "sess-uuid-12345",
  "offset": 0,
  "entry": {
    "type": "agent_output",
    "sub": "spiffe://example.com/agent/A",
    "output_hash": "sha256:abc...",
    "iat": 1700000010,
    "intent_digest": "sha256:...",
    "intent_sig": "eyJ..."
  }
}
¶

5.3. Merkle Tree Construction

The merkle tree is constructed from ordered log entries. Leaf nodes are the SHA-256 hashes of canonically serialized intent chain entries. Internal nodes are the SHA-256 hash of the concatenation of their two child hashes. When a level has an odd number of nodes, the last node is promoted to the next level.¶

                    intent_root (in JWT)
                          |
                +---------+---------+
                |                   |
          Hash(0-2)              Hash(3-5)
                |                   |
        +-------+-------+   +-------+-------+
        |               |   |               |
     Hash(0-1)      Hash(2) Hash(3-4)    Hash(5)
        |               |       |           |
    +---+---+           |   +---+---+       |
    |       |           |   |       |       |
 Entry0  Entry1     Entry2 Entry3 Entry4  Entry5
 (A out) (F1-nd)   (F2-det)(B out)(F3-det)(C out)
¶

Construction Algorithm:¶

def compute_merkle_root(entries):
    if len(entries) == 0:
        return None

    # Compute leaf hashes
    hashes = [sha256(canonical_json(entry)) for entry in entries]

    # Build tree bottom-up
    while len(hashes) > 1:
        next_level = []
        for i in range(0, len(hashes), 2):
            if i + 1 < len(hashes):
                combined = sha256(hashes[i] + hashes[i+1])
            else:
                combined = hashes[i]  # Odd node promoted
            next_level.append(combined)
        hashes = next_level

    return hashes[0]
¶

7. Verification Procedures

7.1. Full Chain Verification

To verify the complete intent chain:¶

  1. Extract intent_root and intent_registry from token.¶
  2. Fetch full intent chain from registry.¶

  3. Verify each entry:¶

    1. For agent_output: verify intent_sig over intent_digest.¶
    2. For non_deterministic_filter: verify intent_sig over intent_digest.¶
    3. For deterministic_filter: optionally re-derive output from input + rules.¶
  4. Rebuild merkle tree from entries.¶
  5. Compare computed root with intent_root.¶
  6. If match, chain is valid.¶

7.2. Single Entry Verification (Merkle Proof)

To verify a single entry without fetching the full chain:¶

  1. Extract intent_root from token.¶
  2. Request merkle proof for entry[i] from registry.¶

  3. Registry returns:¶

    • entry[i]¶
    • Sibling hashes on path to root¶
  4. Compute hash of entry[i].¶
  5. Compute path to root using sibling hashes.¶
  6. Compare computed root with intent_root.¶
  7. If match, entry is in the chain.¶

Proof Size: O(log n) where n is the number of entries.¶

Example Merkle Proof:¶

{
  "entry": {
    "type": "agent_output",
    "sub": "spiffe://example.com/agent/A",
    "output_hash": "sha256:abc...",
    "iat": 1700000010
  },
  "proof": {
    "index": 0,
    "siblings": [
      {"position": "right", "hash": "sha256:111..."},
      {"position": "right", "hash": "sha256:222..."},
      {"position": "left", "hash": "sha256:333..."}
    ]
  }
}
¶

9. Policy Enforcement

9.1. Policy Examples

9.1.1. Require All Outputs Filtered

Every agent output must be followed by at least one filter:¶

require_filtered_outputs {
    intent_chain := get_intent_chain(input.intent_root)

    agent_outputs := [i |
        intent_chain[i].type == "agent_output"]

    every i in agent_outputs {
        # Next entry must be a filter (if not last)
        i < count(intent_chain) - 1
        intent_chain[i + 1].type != "agent_output"
    }
}
¶

9.1.2. Require Non-Deterministic Filter for AI Outputs

AI agent outputs must pass through an AI guardrail:¶

require_ai_guardrail {
    intent_chain := get_intent_chain(input.intent_root)

    every i, entry in intent_chain {
        entry.type == "agent_output" implies {
            some j
            j > i
            intent_chain[j].type ==
                "non_deterministic_filter"
            intent_chain[j].model_info.model ==
                "llama-guard-3"
        }
    }
}

9.1.3. Verify Specific Transformation Applied

Sensitive fields must be sanitized:¶

require_pii_redaction {
    intent_chain := get_intent_chain(input.intent_root)

    some i
    intent_chain[i].type == "deterministic_filter"
    intent_chain[i].rule_id == "pii-redaction-v1"

10. Security Considerations

10.1. STRIDE Analysis

Table 11
Threat Mitigation Mechanism
Spoofing Cryptographic identity Actor chain, SPIFFE IDs
Tampering Merkle tree integrity Append-only logs, merkle root in JWT
Repudiation Signed outputs intent_sig on agent and filter entries
Information Disclosure Selective disclosure SD-JWT, content hashes not content
Denial of Service Session lifecycle Expiry, rate limits
Elevation of Privilege Scope attenuation Actor chain, policy enforcement

10.2. Signing Requirements by Entry Type

Table 12
Entry Type Required Signatures Rationale
agent_output intent_sig over intent_digest (REQUIRED) Non-reproducible; signs input_hash + output_hash to prove what was received and produced
non_deterministic_filter intent_sig over intent_digest (REQUIRED) Non-reproducible; signs input_hash + output_hash to prove transformation
deterministic_filter intent_sig over intent_digest (REQUIRED) Signs input_hash + rule_id; output_hash recorded. Output reproducible from input + rules but signature required for non-repudiation

10.3. Replay Protection

Each intent chain entry includes an iat (issued-at) timestamp. The session-scoped partitioning of the intent registry prevents cross-session replay. Additionally, the merkle root is bound to the JWT via the intent_root claim, and the JWT itself carries exp and jti claims, providing token-level replay protection.¶

10.4. Chain Integrity

The merkle tree structure provides tamper evidence for the intent chain. Any modification to an entry changes its leaf hash, which propagates up the tree, changing the merkle root. Since the merkle root is included in the signed JWT, any tampering with the intent chain entries is detectable.¶

The append-only property of the intent registry provides additional protection: entries cannot be deleted or modified after creation.¶

10.5. Credential Isolation

Intent registry entries MUST NOT contain OAuth tokens, bearer credentials, or signing keys. The relationship between tokens and registry entries is one-directional: the token references the registry via the intent_registry URI claim, but the registry MUST NOT store or reference the token. This separation ensures that compromise of the intent registry does not expose bearer credentials that could be used for unauthorized access.¶

11. Privacy Considerations

11.1. Selective Disclosure

The intent chain contains content hashes rather than actual content. This provides provenance without exposing the content itself. When combined with SD-JWT [[!I-D.ietf-oauth-selective-disclosure-jwt]], individual intent chain entries can be selectively disclosed to different verifiers.¶

11.2. Content Hash vs Content Storage

The intent chain stores SHA-256 hashes of content, not the content itself. This design choice provides:¶

  • Provenance without exposure: Verifiers can confirm that specific content was produced without seeing the content.¶
  • Reduced log size: Hashes are fixed-size regardless of content size.¶
  • Privacy by default: Content is not exposed through the intent chain; access to the original content requires separate authorization.¶

12. Implementation Guidance

12.1. Actor Registry Implementation

The actor registry stores mutable state for authorization. Recommended properties:¶

  • Strong consistency for revocation checks¶
  • Indexed by SPIFFE ID for capability lookups¶
  • TTL-based cleanup of expired capabilities¶

12.2. Intent Registry Implementation

The intent registry stores immutable intent chain entries. Recommended properties:¶

  • Append-only log structure (e.g., Kafka, Kinesis, Pulsar)¶
  • Partitioned by session ID for isolation¶
  • Configurable retention period¶
  • Merkle root computation triggered on append or at token exchange time¶

12.3. Scalability Considerations

  • Log Partitioning: Session-based partitioning ensures that intent chains for different sessions are isolated and can be processed in parallel.¶
  • Merkle Root Caching: Computed merkle roots SHOULD be cached to avoid recomputation on every token exchange.¶
  • Proof Materialization: Merkle proofs for recent entries SHOULD be pre-computed and cached for O(1) retrieval.¶

12.4. Operational Recommendations

  • Retention Policy: Intent chain logs SHOULD be retained for the maximum audit window required by the deployment's regulatory environment.¶
  • Monitoring: Operators SHOULD monitor intent chain append latency and merkle root computation time.¶
  • Backup: Intent chain logs SHOULD be replicated across availability zones for durability.¶

13. Design Rationale: Merkle Root in Token

The decision to carry only intent_root in the OAuth token — rather than embedding the full intent chain inline — is driven by concrete engineering and security trade-offs. This section compares the chosen approach against alternatives.¶

13.1. Alternative Approaches

Table 13
Approach Token Size Verification Privacy Selective Verify
A. Full chain in token O(n) — grows per entry Inline, zero latency Poor — all entries exposed All-or-nothing
B. Merkle root in token O(1) — ~64 bytes O(log n) per entry Good — selective disclosure Single-entry proofs
C. Simple hash of chain O(1) — ~64 bytes O(n) — must rehash all Good — external storage Must verify all
D. No provenance in token Zero overhead External lookup Best — nothing in token Any pattern

13.2. Why Merkle Root

Token size: JWTs traverse East-West in every service-to-service call. The actor chain ([[!I-D.draft-mw-spice-actor-chain]]) is embedded inline because it is small (typically 3-5 entries). Intent chains can contain 20-50+ entries (every filter transformation produces an entry). Embedding them inline would bloat tokens beyond practical header size limits in data-plane proxies (e.g., Envoy default 60KB total request headers).¶

Selective verification (O(log n)): With a simple hash (approach C), verifying ANY single entry requires fetching and rehashing the ENTIRE chain. With a Merkle tree, a verifier proves a single entry using O(log n) sibling hashes — the "Merkle proof." This enables tiered verification: medium-risk operations verify only the most recent entry, not the full chain.¶

Cryptographic binding: Approach D (no provenance in token) provides no cryptographic binding between the token and the provenance data. A compromised registry could serve fabricated entries. The root hash in the token acts as a commitment: the token issuer vouches that "the provenance data for this session hashes to this root."¶

Actor chain stays inline: The actor chain uses approach A because: (a) delegation chains are small, (b) every Resource Server needs the full delegation path for policy decisions, and (c) offline verification is essential. Intent chains are different — they are large, and most Resource Servers only need the root for integrity checks.¶

14. Audit Procedures

14.1. Independent Chain Verification

Each chain in the governance framework can be independently verified by a post-hoc auditor. Each chain answers a distinct question:¶

14.1.1. Actor Chain Audit

Question: Was the delegation path legitimate?¶

  1. Retrieve archived JWT token.¶
  2. Verify JWT outer signature (AS key).¶
  3. For each actor_chain entry (Self-Attested Mode): recompute chain_digest from preceding entries; verify chain_sig against actor's public key.¶
  4. Verify Proof of Residency (PoR) if present.¶
  5. Check actor identities against authorized allowlists.¶

14.1.2. Intent Chain Audit

Question: Was the content transformation history intact?¶

  1. Extract intent_root and intent_registry from token.¶
  2. Fetch ALL entries from intent registry for the session.¶
  3. For each entry: verify intent_sig (for non-deterministic entries); for deterministic filters, optionally re-derive output from input + rules to confirm reproducibility.¶
  4. Rebuild merkle tree from all entries.¶
  5. Compare computed root with intent_root in token.¶
  6. If match: no entries were added, removed, or modified.¶

14.1.3. Inference Chain Audit

Question: Did the claimed AI models actually perform the inference?¶

  1. Extract inference_root and inference_registry from token.¶
  2. Fetch ALL entries from inference registry for the session.¶
  3. For each ZKML entry: fetch verification key; run ZKP verifier.¶
  4. For each TEE entry: validate platform certificate chain; verify quote signature; verify enclave_measurement and report_data binding.¶
  5. Rebuild merkle tree; compare computed root with inference_root.¶

14.2. Cross-Chain Binding Checks

When auditing multiple chains together, the auditor performs cross-chain binding checks that are not possible with any single chain:¶

14.2.1. Actor to Intent Binding

For each intent chain entry of type agent_output: verify that entry.sub appears in actor_chain. Verify that entry.iat falls within the actor's active window. A mismatch indicates an unregistered agent produced content.¶

14.2.2. Intent to Inference Binding

For each inference chain entry: retrieve the referenced intent entry via intent_entry_ref. Verify that inference.output_hash matches the intent entry's output_hash. A mismatch indicates the inference proof does not correspond to the recorded content.¶

14.2.3. Actor to Inference Binding

For each inference chain entry: verify that inference.sub appears in actor_chain. For TEE proofs, verify that inference.por_ref matches the actor's PoR in the actor chain (same hardware). A mismatch indicates a computational proof from an unauthorized agent or unexpected hardware.¶

14.3. Audit Modes

Table 14
Audit Mode Chains Verified Question Answered
Actor only Actor Was the delegation path authorized?
Intent only Intent Was content tampered with?
Inference only Inference Was the claimed model used?
Actor + Intent Cross-binding Did authorized agents produce the content?
Intent + Inference Cross-binding Is every output backed by a computational proof?
Full audit All three + cross-bindings Complete governance verification

Full three-chain audit is RECOMMENDED for regulatory submissions, dispute resolution, and post-breach forensic analysis.¶

15. Future Work: Inference Verifiability

15.1. The Computational Integrity Gap

The intent chain addresses content provenance — proving WHAT each agent produced and HOW filters transformed it. However, it does not address a deeper question: was the claimed AI model actually used to produce the output?¶

An agent can claim to use a high-capability model (e.g., GPT-4, Claude 3.5) while actually running a cheaper, less safe model. The intent chain captures the output hash, but cannot verify the computational process that produced it. This "Model Masquerading" threat represents a gap in the STRIDE coverage:¶

Table 15
Threat Current Coverage Gap
Content Tampering Intent chain (merkle root) Covered
Output Repudiation Intent chain (signed entries) Covered
Computational Spoofing Not covered Model masquerading

15.2. The Inference Root Concept

The companion specification ([[!I-D.draft-mw-spice-inference-chain]]) defines an inference_root claim that provides cryptographic proof of computational integrity. This creates a three-axis "Truth Stack":¶

  • actor_chain: WHO performed the delegation (identity provenance)¶
  • intent_root: WHAT was produced and transformed (content provenance)¶
  • inference_root: HOW the computation was performed (computational provenance)¶

Two mechanisms are defined for generating inference proofs:¶

  1. ZKML (Zero-Knowledge Machine Learning): Cryptographic proofs (SNARKs/STARKs) that a specific model with specific weights produced a given output. Provides mathematical certainty without hardware trust assumptions, but currently limited to small models due to computational overhead.¶

  2. TEE Attestation (Confidential Computing): Hardware-signed quotes from Trusted Execution Environments (e.g., NVIDIA H100 TEE, Intel TDX) proving that a specific model ran within a verified enclave. Works with production-scale LLMs today, but requires trust in the hardware manufacturer's root of trust. This approach aligns with the Proof of Residency mechanism defined in [[!I-D.draft-mw-spice-transitive-attestation]].¶

The inference_root follows the same Merkle tree pattern as the intent_root, with proof entries stored in a dedicated inference registry and only the root hash included in the OAuth token.¶

15.3. Relation to This Specification

The intent chain and inference chain are complementary but independent:¶

  • The intent chain MAY be deployed without inference verification (content provenance without computational proof).¶
  • The inference chain SHOULD reference intent chain entries to bind computational proofs to specific content outputs.¶
  • Together, they provide end-to-end verifiability from model execution through content transformation to final output.¶
